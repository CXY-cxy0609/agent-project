# RAG Chunking 优化方案（语义优先 + 结构保护）

## 1. 背景与目标

当前分块策略是“语义优先，固定大小兜底”：

- Markdown：先按标题边界切分，超长后滑动窗口。
- Plain text：先按空行分段，再拼接到上限，必要时切分。

这套策略可用，但在复杂文档（表格、代码块、公式、OCR 文本）上存在隐性质量风险。本文档目标是：

- 保留现有策略的稳定性与简单性；
- 提升结构化内容（表格/代码/列表）的完整性；
- 让 chunk 大小控制更可预测（真正 token 级）；
- 建立可观测指标，避免“改了但不知道是否更好”。

## 2. 现状与隐性问题

### 2.1 表格与结构化片段易被切碎

- Markdown 表格在超长后进入按词滑窗，可能拆散表头与数据行。
- PDF/OCR 产物中的“伪表格”通常被当普通段落，语义对齐丢失。

影响：召回后上下文可读性下降，rerank 也难以恢复结构信息。

### 2.2 代码块与公式块缺乏保护

- 标题切分未显式规避 fenced code block。
- 长代码块被按词切，容易破坏语义完整单元。

影响：技术文档问答时，引用片段不完整，答案正确率下降。

### 2.3 overlap 语义与实现不一致

- 配置是 token 维度 overlap；
- 实现中步进基于 words 且有额外比例折算，实际 overlap 偏离配置。

影响：chunk 之间冗余度和召回覆盖不稳定，调参不可预期。

### 2.4 短块合并策略不完整

- 只优先向后合并，尾部短块可能残留；
- metadata 合并保留不完整，可能丢失章节上下文。

影响：产生低质量碎片 chunk，影响召回排序与可解释性。

### 2.5 token 估算在退化路径误差较大

- tiktoken 不可用时使用字符近似；
- 中英混排时误差明显。

影响：max/min token 控制不稳定，召回质量波动。

## 3. 优化原则

1. **结构优先**：先保护结构化语义单元（表格、代码块、列表、标题段）。
2. **token 一致性**：切分、步进、重叠都在 token 语义下定义。
3. **最小侵入**：不推翻现有 pipeline，优先在 `chunker.py` 增量改造。
4. **可观测性**：引入离线评估和运行时统计。

## 4. 目标方案（建议实现）

## 4.1 分层切分器（推荐）

将 chunking 分成三层：

- **Layer A: 结构块识别（Block Parser）**
  - 识别：标题段、表格块、代码块、列表块、普通段落块。
  - 输出统一 block 序列，每个 block 带 `block_type` 与原始边界。

- **Layer B: 语义合并（Section Assembler）**
  - 优先在同类型或兼容类型 block 内合并到 `target_tokens`。
  - 禁止跨越“不可拆结构块”（如完整表格、完整代码块）进行粗暴切断。

- **Layer C: token 滑窗兜底（Window Fallback）**
  - 仅对超长单块（极长代码/表格）进行 token 级窗口切分；
  - overlap 严格按 `overlap_tokens = int(max_tokens * overlap_ratio)` 计算；
  - 步进以 token 为单位，避免 word 近似偏差。

## 4.2 结构保护规则

- **表格**
  - Markdown 表格按连续行识别为单 block；
  - 超长切分时优先“按行”切，并在每个子块保留表头。

- **代码块**
  - fenced code block（```）整体识别；
  - 超长时按“函数/空行/行数阈值”切，避免任意截断。

- **列表**
  - 保持列表项完整，不在 item 中间切分。

- **标题**
  - 章节 metadata 继承严格化：每个 chunk 保留最近有效标题链。

## 4.3 合并与去碎片规则

- short chunk 采用“双向合并”策略：优先后并，无法后并则前并。
- 合并 metadata 改为“可追溯并集”：
  - 保留主要章节字段；
  - 追加来源范围（如起止行/页）。

## 4.4 Plain text（PDF/OCR）增强

- 引入轻量规则识别“伪表格”与“段内断行”：
  - 连续多行包含规则分隔符（`|`, 多空格对齐）视为表格候选；
  - 过短换行尝试回流为同段，减少 OCR 断裂。
- 段落分组时优先按语义标点与标题样式断开，而非仅空行。

## 5. 分阶段落地计划

### Phase 1（低风险，高收益）

- 修复 token overlap 语义一致性（token 级步进）。
- 增加 fenced code block、markdown table 的基础识别与保护。
- short chunk 双向合并 + metadata 合并修复。

交付：不改对外接口，只改 `chunker.py` 行为。
范围控制：以 2 周 MVP 为目标，不引入完整 block parser，不引入视觉双索引。

### Phase 2（质量强化）

- 增加轻量 block parser（优先标题/表格/代码三类）。
- plain text OCR 规则增强（伪表格识别、断行回流）。
- 评估是否需要视觉双索引（先灰度验证，不默认开启）。

交付：新增内部辅助函数与测试样例集。

### Phase 3（评估与调参）

- 建立离线评估集（技术文档、表格密集文档、OCR 文档）。
- 对比改造前后召回与问答指标。
- 固化默认参数（`max_chunk_size`, `chunk_overlap`, `min_chunk_size`）。
- 决策是否推广高成本链路（基于收益/成本比）。

## 6. 验收指标（必须量化）

- **结构完整率**：表格/代码块被完整保留的比例。
- **碎片率**：`token < min_chunk_size` 的 chunk 占比。
- **平均冗余度**：相邻 chunk 重叠 token 占比与目标偏差。
- **检索效果**：Recall@K / MRR（基于标注问题集）。
- **问答效果**：人工评估正确率、引用片段可读性评分。
- **成本指标**：单文档平均处理成本、每次问答视觉升级成本。
- **时延指标**：P95 首次响应时延、P95 升级链路时延。

建议验收门槛：

- 结构完整率提升 >= 30%；
- 碎片率下降 >= 40%；
- Recall@20 不下降，目标提升 5%+；
- 端到端问答正确率提升或持平且可解释性提升。
- 单文档处理成本增幅 <= 30%；
- P95 首次响应时延增幅 <= 20%。

## 7. 风险与回滚策略

- 风险：规则复杂度上升，可能影响切分吞吐。
- 缓解：通过 feature flag 控制新策略开关（`CHUNKER_V2_ENABLED`）。
- 缓解：高成本链路默认关闭，仅在触发条件满足时开启。
- 回滚：保留旧路径，出现召回回退时可一键切换。

## 8. 建议优先级

若只做一轮迭代，建议优先做：

1. token 级 overlap 一致性修复；
2. 表格/代码块保护；
3. short chunk 双向合并与 metadata 修复。

这三项实现成本较低，但对检索稳定性和答案可读性的提升最直接。

## 8.1 MVP 边界（效果与成本平衡）

MVP 只做“高收益、低耦合”改造：

- 必做：token overlap 一致性、表格/代码块保护、短块双向合并。
- 暂缓：完整多类型 block parser、视觉双索引、复杂结构化重建。
- 原则：先解决明确痛点，再考虑架构级升级。

## 8.2 成本红线（默认约束）

- 不允许对全量 PDF 页面默认执行高成本视觉链路。
- 高成本升级必须满足触发条件（如低文本覆盖率、低置信度命中页）。
- 每次问答限制升级页数（建议 3-5 页），超限走降级策略。
- 所有 OCR/视觉结果必须缓存（`doc_hash + page_hash + pipeline_version`）。

## 8.3 上线门禁（Go/No-Go）

满足以下条件才进入下一阶段：

- 质量门槛：Recall@20 提升 >= 3% 或问答正确率显著提升；
- 成本门槛：单文档成本增幅 <= 20%，P95 首次响应时延增幅 <= 20%；
- 稳定性门槛：核心链路无新增 P1 故障。

停止条件（防过度工程）：

- 若连续两轮评估均出现“效果提升 < 3% 且成本增幅 > 20%”，暂停 Phase 2/3 的高复杂度投入。

## 9. 图片型 PDF 专项方案（效果与成本平衡）

### 9.1 问题定义

针对“页面主要是图片（扫描件、拍照件、图表页）”的 PDF，单纯文本解析会出现：

- 可提取文本极少，导致召回缺失；
- OCR 成本高，整本重处理不经济；
- 图表/流程图等视觉信息难以被纯 OCR 文本表达。

目标是在可控成本下，保证图片型页面的可检索性与回答质量。

### 9.2 总体策略：分层路由 + 延迟精加工

1. **先体检，再处理**
   - 页级检测文本覆盖率、清晰度、疑似图表/表格密度；
   - 将页面标记为 `text_page` / `image_page` / `mixed_page`。

2. **先低成本，再升级**
   - 默认走低成本 OCR（快速抽取）；
   - 仅对低置信度或高价值页面升级高质量视觉模型。

3. **先召回，再精细化**
   - 入库阶段做“够用”的页级抽取；
   - Query 命中后仅对 Top-N 页面做精细 OCR/结构化解析（lazy refine）。

4. **文本索引 + 视觉摘要双通道**
   - OCR 文本进入主向量索引；
   - 图表/流程图等页面额外生成视觉摘要（caption + 关键信息）并索引。

### 9.3 推荐处理流水线（图片型 PDF）

- **Step A: Page Triage（页级分流）**
  - 输出：`page_type`, `text_density`, `quality_score`。
- **Step B: Fast OCR（默认）**
  - 抽取正文、标题、可识别公式（低成本模型）。
- **Step C: Upgrade OCR / Vision（按需）**
  - 触发条件：低置信度、图表密集、用户问题命中该页。
- **Step D: Chunking with Structure**
  - 对 OCR 产物使用增强后的 chunker（表格/代码/列表保护）。
- **Step E: Dual Index**
  - 写入 `text_chunks` 与 `visual_summaries` 两类索引。

### 9.4 成本控制机制（必须配置）

- **预算上限**
  - 设定每文档/每请求最大升级页数（如 3-5 页）；
  - 设定每次问答的视觉处理 token 或调用上限。

- **升级比例**
  - 默认只升级疑难页 Top 20%-30%，其余保留快速路径结果。

- **强缓存**
  - 以 `doc_hash + page_hash + pipeline_version` 做结果缓存；
  - OCR 与视觉摘要分别缓存，避免重复扣费。

- **异步预热**
  - 首次回答走快速结果；
  - 后台异步补齐高质量页，二次查询命中缓存。

### 9.5 质量保障指标（图片专项）

- 图片页可召回率（image-page recall@k）；
- 图表问答正确率（chart QA accuracy）；
- 页级 OCR 升级命中收益（升级页对最终答案贡献度）；
- 单文档平均处理成本、P95 响应时延。

建议目标：

- 图片页召回率提升 >= 20%；
- 图表相关问答正确率提升 >= 15%；
- 单文档处理成本增幅控制在 <= 30%。

## 10. Agent 侧最小改造清单（配合落地）

说明：核心检索和 chunking 优化仍在 RAG 服务实现，Agent 侧主要负责“策略路由与预算控制”。

### 10.1 必须改（MVP）

- `apps/agent/src/tools/file-parser.tool.ts`
  - 增加可选参数：`mode`（`fast|balanced|quality`）、`max_upgrade_pages`、`budget_tokens`；
  - 接收并透传页级质量信号（如 `page_type`, `quality_score`）。

- `apps/agent/src/harness/rag-client/rag-client.ts`
  - `retrieve()` 增加可选参数：`retrieval_mode`（`text_only|hybrid_visual`）与预算约束；
  - cache key 增加策略维度，避免不同策略互相污染缓存。

- `apps/agent/src/agents/qa`（graph/node）
  - 在 OCR -> RAG 之间加入路由节点：
    - 普通文本问题走默认链路；
    - 图像密集/低置信度命中时触发升级链路。

兼容性约束（必须遵守）：

- 所有新参数均为可选，不影响旧请求。
- 默认行为与当前一致（不触发高成本升级）。
- cache key 仅在新策略参数出现时扩展，避免历史缓存失效。

### 10.2 建议改（增强）

- `apps/agent/src/container.ts`
  - 注入策略配置项（预算、升级阈值、默认模式），避免硬编码。

- 可观测性
  - 记录每次问答触发的升级页数、升级耗时、命中收益与成本。

### 10.3 可后置

- Prompt 体系无需结构性改造；
- 仅在 task 模板增加“信息不足时请求补充页/区域”的兜底提示。
